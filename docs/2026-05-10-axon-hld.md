## 1. System Overview

AXon là một web platform nội bộ bao gồm 3 thành phần chính:

| Thành phần | Công nghệ | Vai trò |
|-----------|-----------|---------|
| Frontend | React 18 + TypeScript + Vite + TailwindCSS | SPA giao diện người dùng |
| Backend | Java 21 + Spring Boot 3 | REST API, business logic, xác thực |
| Infrastructure | PostgreSQL, Redis, Docker Volume | Dữ liệu, session/cache, file storage |

Hệ thống tích hợp với 1 external system:
- **CIP/AD** (Samsung Corporate Identity Provider — SSO nội bộ)

```
axon-frontend/

├── Library Module    — Browse, search, filter, sort, trending
├── Detail Module     — Xem chi tiết BP, download, like, feedback

├── MyBPs Module      — Danh sách BP của AX Creator, analytics
```

**State management:** Zustand (auth state), TanStack Query (server state/cache)  
**HTTP client:** Axios với interceptor tự động gắn JWT và refresh token

### 3.2 Backend (Spring Boot)

```
axon-backend/
├── auth/             — CIP/AD provider interface & mock, JWT
├── user/             — User entity, role management
├── bestpractice/     — CRUD, submit workflow, state machine
├── file/             — Upload/download qua Docker volume (local filesystem)
├── management/       — AX Supporter: approve/reject/close, review history
├── interaction/      — Like, download tracking, feedback
├── lookup/           — Job, AI capability, work, department, AI tool APIs
├── analytics/        — BP analytics cho AX Creator
├── dashboard/        — Dashboard stats cho AX Supporter
├── masterdata/       — CRUD Job, WorkCategory, Work, AICapability (Admin)
├── aiinsight/        — Static content: 5 AI capability classifications
└── notification/     — Email notification service
```

### 3.3 Database (PostgreSQL)

**Lookup tables (5):** `jobs`, `ai_capabilities`, `work_categories`, `works`, `departments`

**Core tables (3):** `users`, `best_practices`, `bp_files`

**Junction tables (3):** `bp_creators`, `bp_jobs`, `bp_ai_capabilities`

**Interaction tables (4):** `bp_likes`, `bp_feedback`, `bp_downloads`, `bp_reviews`

### 3.4 Cache (Redis)

- Session / refresh token store (TTL 7d)
- Rate limiting counters
- Dashboard stats cache (TTL 15 phút)

### 3.5 File Store (Docker Volume)

- Mount path trong container: `/app/uploads`
- Host mount: `./uploads:/app/uploads` trong Docker Compose
- File key pattern: `{best_practice_id}/{uuid}_{original_filename}`
- Download: Backend stream file từ volume, kiểm tra JWT + quyền trước khi serve
- Chỉ áp dụng với BP type TOOL và EXTENSION

## 4. Key Design Decisions

### 4.1 CIP/AD Pluggable Architecture

CIP/AD là provider chính nhưng thiết kế theo interface để linh hoạt:

```
SSOProvider (interface)
  ├── MockSSOProvider     (dev/test — trả về hardcoded user)
  └── CIPADProvider       (production — Samsung CIP/AD)
```

Spring profile `dev` dùng MockSSOProvider. Production inject CIPADProvider.

### 4.2 BP Status State Machine

```
REQUESTED ─── approve ──► PUBLISHED ─── close ──► REJECTED
    ▲                                                  │
    └──────────── Creator edit + resubmit ─────────────┘
    │
    ◄── reject ── REQUESTED
```

Transition rules:
- Chỉ AX Supporter/Admin mới được approve / reject / close
- AX Supporter không thể approve BP mà mình là creator
- **3 trạng thái:** REQUESTED / PUBLISHED / REJECTED (không có CLOSED)
- Close = `PUBLISHED → REJECTED`; creator vẫn có thể edit và resubmit
- Edit PUBLISHED BP:
  - Sửa file/URL cài đặt → auto `REQUESTED` (BP ẩn)
  - Không sửa file/URL → giữ `PUBLISHED`

### 4.3 File Download Security

Files không bao giờ public. Mọi download đều qua:
1. Backend kiểm tra JWT + quyền xem BP (must be PUBLISHED hoặc owner/supporter)
2. Backend stream file từ Docker volume (`/app/uploads/{bpId}/{filename}`)
3. Backend ghi log DOWNLOAD (bp_downloads table) — async
4. Client nhận file binary trực tiếp (không redirect)

Không dùng pre-signed URL — không còn MinIO.

### 4.4 Key Field Visibility

Trường `key_value` trong best_practices là nhạy cảm:
- **Ẩn** với User thông thường (không trả về trong API response)
- **Hiển thị** với: AX Creator (là owner/creator của BP), AX Supporter, Admin
- Backend kiểm tra role và ownership trước khi include field trong response

### 4.5 Like Mechanism

- Toggle: like lần 1 → thêm; like lần 2 → bỏ (unlike)
- Unique constraint `(bp_id, user_id)` trong table `bp_likes`
- `like_count` trong `best_practices` được update atomic khi toggle
- API trả về cả `like_count` và `is_liked_by_current_user`

### 4.6 Multiple Creators

- Một BP có thể có nhiều creator qua junction table `bp_creators`
- Card hiển thị: tên creator đầu tiên + "+N others" nếu N > 1
- Tất cả creator đều có thể edit BP (nếu status cho phép)
- Self-approve check: AX Supporter bị chặn nếu user_id của họ có trong `bp_creators` của BP đó

### 4.7 Taxonomy Management

Các bảng lookup (job, ai_capability, work_category, work, department) là master data:
- Admin quản lý qua Admin API
- Frontend dùng để populate dropdowns khi đăng ký BP và filter
- Seed data ban đầu theo giá trị mẫu từ requirements

## 5. Security Architecture

```
Request flow:
Browser → [JWT in Authorization header] → Spring Security Filter
  → JwtAuthenticationFilter (validate JWT)
  → SecurityContext (set current user + role)
  → Controller → Service (check role + ownership)
  → Response (mask key_value nếu không đủ quyền)
```

**Token flow:**
```
CIP/AD Login:
  Browser → /auth/login → redirect to CIP/AD
  CIP/AD  → /auth/callback (with code)
  Backend → exchange code → get user info → upsert user → issue JWT pair
  JWT: { access_token (15m), refresh_token (7d) }

Token refresh:
  Browser → /auth/refresh (with refresh_token)
  Backend → validate refresh_token in Redis → issue new access_token
```

### 5.1 Security NFRs (P11 deferred)

> P0–P10: chỉ enforce RBAC + OWASP Top 10 cơ bản (SQL injection, XSS, CSRF). Mock auth, dev data không phải PII thật. Mọi items dưới đây implement ở P11.

| Concern | Approach |
|---------|----------|
| MFA cho ADMIN | Delegate cho CIP/AD MFA flow; backend verify `amr` claim chứa `mfa` cho ADMIN session |
| Encryption at rest | AES-256 cho PII fields (email, cip_id, department) qua JPA `@Convert(converter = AesGcmStringConverter.class)`; key từ env var `PII_ENCRYPTION_KEY` |
| Encryption in transit | TLS 1.2+ enforce ở ingress (Nginx hoặc K8s Ingress); app nhận traffic plain HTTP nội bộ |
| PII audit log | `audit_logs` table + AOP aspect quanh user-PII access methods; retention 1 tháng (scheduled cleanup job) |
| PII masking in logs | Logback `MaskingPatternLayout` filter cho email/cipId; `GlobalExceptionHandler` redact PII trước khi serialize error response |
| Session timeout 30m | Access token TTL = 30 phút; FE auto-refresh trước khi hết hạn |

## 10. Auth Strategy

**Development (P0–P10):** `MockSSOProvider` — 4 hardcoded users để test nghiệp vụ theo role:
- `user1` → USER role
- `creator1` → AX_CREATOR role
- `supporter1` → AX_SUPPORTER role
- `admin1` → ADMIN role

**Production (P11):** Samsung CIP/AD OAuth 2.0/SSO + full Security NFR rollout:
- Browser redirect → CIP/AD login page
- Callback → backend exchange code → get user info (name, email, department/SRV Group)
- Backend upsert user → issue JWT pair (access 30m, refresh 7d)
- Department string tự động lấy từ CIP/AD profile
- **MFA enforcement cho ADMIN role** (qua CIP/AD `amr` claim)
- **PII encryption at rest** cho email/cipId/department (AES-256 via JPA converters)
- **PII audit logging service** ghi `audit_logs` row cho mọi thao tác PII; retention 1 tháng
- **Log + error PII masking** (Logback filter + GlobalExceptionHandler sanitization)
- **TLS 1.2+** enforce ở ingress layer (ngoài app)

```
SSOProvider (interface)
  ├── MockSSOProvider  (dev — hardcoded users, inject via Spring profile "dev")
  └── CIPADProvider    (prod — Samsung CIP/AD OAuth 2.0)
```

Security scope: P0–P10 chỉ RBAC + OWASP Top 10 cơ bản. P11 thêm SSO/MFA/encryption/audit/masking (xem §5.1).
